# Production Runbook

## Purpose

This runbook defines production operations for the WhatsApp CRM platform, including alert response, escalation, and recovery procedures.

## Service Topology

- Frontend: Next.js application
- Backend API: NestJS service (webhooks + business logic)
- Worker: queue processor (BullMQ)
- Database: PostgreSQL
- Cache/Queue: Redis

## Core SLO Targets

- API uptime: >= 99.5%
- Webhook success rate: >= 99%
- WhatsApp send success rate: >= 98%
- P1 first response: <= 30 minutes

## Monitoring Checklist

Run this checklist at start of day and during incidents.

1. Backend availability
- Verify health endpoint returns 200.
- Confirm average latency is within acceptable threshold.

2. Webhook integrity
- Shopify webhook delivery errors are below threshold.
- WhatsApp webhook signature failures are zero or expected during testing.

3. Queue health
- Queue depth is below operational threshold.
- No worker restart loop or stuck jobs.

4. WhatsApp integration health
- Token health status is not error.
- Token expiry is not within emergency window.
- Phone quality rating alerts are reviewed (green/yellow/red).

5. Data safety
- Backup job ran successfully.
- Last restore drill evidence is available and recent.

## Alert Severity Matrix

- P1: Revenue-impacting outage, webhook processing down, message sending fully down.
- P2: Partial degradation, increased failures, queue buildup, elevated latency.
- P3: Non-critical bug, one-off tenant issue, documentation gap.

## Immediate Response Playbooks

### 1) Webhook Failures Spike

Symptoms:
- Sudden increase in webhook processing errors
- Order updates stop appearing

Actions:
1. Check backend uptime and recent deploys.
2. Validate webhook signature configuration and tokens.
3. Inspect request logs for malformed payloads and auth issues.
4. If platform outage is external (Meta/Shopify), switch status to degraded and communicate ETA.

### 2) WhatsApp Send Failures Spike

Symptoms:
- High failed message ratio
- Frequent 429/5xx from provider

Actions:
1. Confirm retry/backoff is active.
2. Check provider status and rate limits.
3. Validate token validity and integration health.
4. Reduce non-critical sends if quality or rate limits are degrading.

### 3) Queue Backlog Growth

Symptoms:
- Queue depth continuously rising
- Delayed automations

Actions:
1. Check worker process health and logs.
2. Scale worker replicas.
3. Confirm Redis and DB are healthy.
4. Reprocess dead-letter or failed jobs as needed.

### 4) Phone Quality Rating Degraded (YELLOW/RED)

Symptoms:
- Quality alert generated by daily checks or webhook updates

Actions:
1. Identify affected workspace and phone number.
2. Review recent campaign/template traffic volume.
3. Pause or reduce risky marketing sends.
4. Ensure opt-in and content quality compliance.
5. Monitor rating trend until recovery.

## Escalation Path

1. On-call engineer triages and classifies incident.
2. Founder/owner notified for P1 within 10 minutes.
3. Customer communication initiated for P1 and major P2.
4. Post-incident review completed within 48 hours.

## Evidence and Audit

For each P1/P2 incident, capture:
- Start and end timestamps
- Root cause
- Workaround and permanent fix
- Impacted tenants
- Follow-up tasks with owners

Store records in Docs/Compliance/Backup-Drills and incident documentation files.

## Weekly Ops Review

- Review top error categories
- Review webhook success trends
- Review quality alerts and affected tenants
- Validate backup and restore readiness
- Update this runbook when gaps are discovered
